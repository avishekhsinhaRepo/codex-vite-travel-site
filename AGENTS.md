# Repository Guidelines

## Project Structure & Module Organization
This is a lightweight Vite app with a single-page travel booking UI.

- `index.html`: app shell, Bootstrap/CDN imports, and section markup.
- `src/main.js`: mock data, filtering logic, booking flow, and DOM event wiring.
- `src/style.css`: design tokens (`:root` variables), layout, and component styles.
- `dist/`: production output generated by Vite (`npm run build`).

If new features increase complexity, split logic into `src/features/<feature>.js` modules and keep DOM selectors/events close to their feature code.

## Build, Test, and Development Commands
- `npm install`: install project dependencies.
- `npm run dev`: start local development server with hot reload (default `http://localhost:5173`).
- `npm run build`: create optimized production bundle in `dist/`.
- `npm run preview`: serve `dist/` locally to verify production behavior.

## Coding Style & Naming Conventions
- Use 2-space indentation in JS and CSS.
- Follow existing JS style: `const`/`let`, arrow functions, semicolons, and double quotes.
- Use `camelCase` for variables/functions (`handleBookingSubmit`, `buildResultMeta`).
- Use `kebab-case` for CSS class names (`hero-section`, `result-card`).
- Prefer reusing existing CSS variables in `:root` before adding new color values.
- Keep functions focused and side effects explicit (render/update steps in dedicated helpers).

## Testing Guidelines
No automated test framework is configured yet. Validate changes with manual checks before opening a PR:

- Run `npm run build` to catch bundling/runtime issues.
- Smoke-test Flights, Buses, and Holidays search flows.
- Verify booking modal validation, confirmation rendering, and responsive layout at mobile/desktop widths.

When adding tests, place them under `src/__tests__/` and use `*.test.js` naming.

## Commit & Pull Request Guidelines
- Use short, imperative commit messages (example: `Fix hero layering for interactive carousel content`).
- Keep commits scoped to one logical change; avoid mixing refactors with feature work.
- PRs should include a clear summary of user-visible changes.
- PRs should include a linked issue/ticket when available.
- PRs should include manual test notes (`npm run build`, flows verified).
- PRs should include before/after screenshots or a short recording for UI changes.
